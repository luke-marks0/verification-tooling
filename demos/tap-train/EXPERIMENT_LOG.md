# Experiment log: tap-train
Started 2026-05-31. See demos/tap-train/plan.md.

- 2026-05-31: Initial build — stacked on PR #19 (`worktree-tap-protocol`). Same 4-server tap-protocol topology adapted to deterministic LoRA training. Workload swapped from OpenAI `/v1/chat/completions` to `workflows.deterministic_lora_training.train_once`, with the workflow refactored to accept `cfg` and `dataset` kwargs (backwards-compatible: defaults to module-level `TRAINING_CONFIG` and `_benign_dataset`); also exposed `hash_adapter_dir` (public alias) and `benign_arithmetic_dataset` for the dataset builder. Wire types in `servers/envelope.py`: `TrainRequest{base_model, weights_revision, lora, hp, dataset}` and `TrainResponse{adapter_digest, final_loss, loss_trajectory, n_steps, n_params_trainable}`; `SignedEnvelope`/`sign`/`verify`/`next_id` copied verbatim from `demos/tap-protocol/servers/envelope.py` with the same hardcoded 32-byte HMAC_KEY. Dataset travels as `DatasetSpec{builder: "benign_arithmetic", num_examples, seed}`; both Host and Recomp resolve the builder name to the same workflow helper, so the dataset is byte-identical without crossing the wire. Single H100, two python processes per cluster, but training is serialized via a per-cluster `STATE.train_lock` — backward-pass memory is too spiky to run two LoRA fine-tunes concurrently on one H100. Tap's `_async_verify` thread uses a 10-minute timeout because re-training takes minutes (vs seconds for inference). Host Cluster adds `GET /adapter/<digest>` that streams a `tar.gz` of the adapter directory — the response envelope carries only the digest, so the client can fetch the actual safetensors after the digest arrives. 21 unit tests under `tests/unit/test_tap_train_envelope.py`: sign/verify round-trips on TrainRequest and TrainResponse envelopes, tamper detection (signature/payload/id), canonical-JSON stability across dict insertion order, list-order asymmetry for `lora.target_modules` (lists are semantically ordered, so reordered lists MUST produce different signatures), id monotonicity single + threaded — all pass under `.venv/bin/python3 -m unittest tests.unit.test_tap_train_envelope`. CPU mock smoke: happy path with empty `{}` request → server returns `{"adapter_digest": "sha256:3c91...bd2", "final_loss": 0.0, "loss_trajectory": [], "n_steps": 32, "n_params_trainable": 0}`, no `alarm.jsonl`, tap logs `verify verdict for id=1: {'is_verified': True}`; forced mismatch via `recomp --mock-output-override "sha256:deadbeef..."` → one canonical-JSON alarm line containing both digests, full `train_request_summary`, `first_divergent_step: -1` (trajectories empty in mock mode), `reason: "digest_mismatch"`; signature tampering (hand-crafted envelope with one flipped hex char) → HTTP 401 from tap, alarm count unchanged (request never reached recomp). The mock digest is `"sha256:" + sha256(canonical_json_bytes(TrainRequest.model_dump()))` — depends only on the request payload, so two clusters with the same envelope produce the same mock digest by construction. Real-GPU run deferred to the launcher; no GPU available in this implementation session. Open follow-ups: extract `envelope.py` into a shared module so the inference and training demos stop diverging; add a streaming logger on the Host Cluster so an operator can tail training loss live without polling.

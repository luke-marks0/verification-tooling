# Experiment log: tap-protocol
Started 2026-05-28. See demos/tap-protocol/plan.md.

- 2026-05-28: Design v0 — Drafted the 4-server inline-tap verification protocol. Topology: Gateway → Tap → Host Cluster, with Tap forking a fire-and-forget copy to a Recomputation Cluster. Both the Host and Recomp Clusters reuse `modules/inference/server/main.py` (deterministic vLLM) with `modules/inference/manifests/qwen3-1.7b.manifest.json`; two vLLMs coexist on one H100 at `gpu_memory_utilization=0.45` each. `/verify` does full re-inference and bitwise UTF-8 compare against the tapped Host output; mismatches append to `alarm.jsonl`. HMAC-SHA256 SignedEnvelope with a hardcoded shared key (constant in `servers/envelope.py`); Gateway-assigned monotonic id ties request↔response↔verification. Code lives in `demos/tap-protocol/`; the headline deliverable is `scripts/launch_vast.sh` which provisions an H100, rsyncs the worktree, starts the four servers, and prints the Gateway URL. Worktree is `worktree-tap-protocol`.
- 2026-05-28: Adversarial review + plan v1 — Ran an adversarial reviewer over plan.md. Critical findings that drove changes: the repo's `qwen3-1.7b.manifest.json` pins `TRITON_ATTN` (not FLASH_ATTN required by c3), `gpu_memory_utilization=0.9` (no CLI override in `_build_vllm_cmd`), `max_num_seqs=256 + max_model_len=8192` (KV cache too big for two coexisting instances), `cuda_launch_blocking=true` (serializes everything), and GH200 hardware_profile. Fixed by introducing a demo-local `demos/tap-protocol/qwen3-1.7b-tap.manifest.json` with FLASH_ATTN, gpu_memory_utilization=0.40, max_num_seqs=8, max_model_len=2048, cuda_launch_blocking=false, H100 hardware. Also threaded `max_tokens` through the envelope (was hardcoded in two files — refactor hazard). Launcher fixes: dynamic ghcr entrypoint resolution, valid rsync `-e "ssh -p $PORT"` syntax (was `IP:$PORT:path`), `setsid` + redirected stdin to survive ssh disconnect, `CUDA_VISIBLE_DEVICES=0` pin, `RUNNER_MODEL_PATH` set after weights prefetch so vLLM doesn't race on the HF cache, sequential cluster start so KV-cache sizing isn't racy. Added a post-Popen warm-up step (one canned inference) before each cluster flips its own /health to 200 — sidesteps vLLM's `engine not initialized` window. Owned the HMAC threat-model honestly in §10 (integrity for the localhost channel only; not auth, not anti-replay). `alarm.jsonl` opened append-only.
- 2026-05-28: First successful end-to-end run on a real H100 (vast.ai instance 38256025, offer 29019359, H100 SXM 80 GB HBM3, Czechia). The first launch_vast.sh run failed in a sequence of nix-image realities the CPU smoke couldn't have surfaced: (1) `vastai create` returned a Python dict repr `'new_contract': N` with single quotes but the launcher's regex looked for `"new_contract":` (double-quoted JSON) — fixed to accept both; (2) `rsync` is not on the nix vast-test image — replaced with `tar -czf` locally + `scp -P PORT` (uppercase, not `-p` which means preserve-mtime in scp) + `python3 -m tarfile -e` on the remote; (3) `pip`/`ensurepip` missing — but `huggingface_hub` is already bundled with vllm, so `from huggingface_hub import snapshot_download` works directly with no install step; (4) `setsid` missing — replaced with `( nohup CMD </dev/null >LOG 2>&1 & )` subshell double-fork (subshell exits immediately, child reparents to init); (5) `curl` missing on the nix image — `wait_healthy` in start_servers.sh now probes with a python3 urllib heredoc, not curl; (6) the standard bash arithmetic pitfall — `(( i++ ))` returns 1 when i=0 (false in arithmetic context) and under `set -e` aborts the script silently on the very first loop iteration; replaced with `i=$((i+1))`. The actual code bug surfaced only on GPU: with `RUNNER_MODEL_PATH` set, vLLM serves under the snapshot path (not "Qwen/Qwen3-1.7B"), so the shim's hardcoded MODEL_ID got 404'd on /v1/chat/completions — added `_resolve_model_id()` to both shims that GETs /v1/models at boot and stashes the actual served name in `STATE.model_id`. One non-code lesson: SIGKILL on a running vLLM leaks the CUDA context's GPU allocation (free dropped from 79 → 14 GiB across two restart cycles), requiring `vastai reboot instance` to reclaim — when iterating, use SIGTERM and wait for the shim's signal handler to terminate vLLM cleanly. Final result on the real H100: gateway at `http://80.188.223.202:18032` (container port 8000 maps to host port 18032 — NOT 8000 directly; pull from `vastai show instance --raw`'s `ports."8000/tcp"[0].HostPort`); two real prompts ("The capital of France is" and "Write one line of Python that prints hello world:") returned Qwen3-1.7B `<think>...` completions; both were async-verified by the Recomp Cluster's independent re-inference (host_cluster vLLM on port 8022, recomp_cluster vLLM on port 8032, both with FLASH_ATTN + enforce_eager + VLLM_BATCH_INVARIANT + CUBLAS_WORKSPACE_CONFIG=:4096:8 + seed=42 + temperature=0 + gpu_memory_utilization=0.40); each produced bitwise-identical output strings; `alarm.jsonl` never created; tap.log shows `verify verdict for id=1: {'is_verified': True}`. The c3 determinism contract holds across two coexisting vLLM processes on one H100, end-to-end through the SignedEnvelope wire protocol, demo accomplished.
- 2026-05-28: Reviewer + tester follow-ups applied — Adversarial code reviewer reported zero blockers ("ship it"); CPU integration tester reported 5/5 smoke scenarios pass (happy path, forced mismatch, signature tampering 401, id correlation, bad input). Applied the high-value should-fix items: cluster shims `os._exit(1)` on warm-up failure so `start_servers.sh` can't loop forever against a dead process (host_cluster.py, recomp_cluster.py); threaded SIGTERM handlers (the previous code called `server.shutdown()` from the signal-handler thread while main was parked in `serve_forever`, which can deadlock — now spawns a daemon thread for the shutdown call, all four servers; shutdown of the 4-server stack went from >1s to 741ms in re-smoke). Gateway `--host` default flipped from `0.0.0.0` to `127.0.0.1`; `start_servers.sh` now passes `--host 0.0.0.0` explicitly so the public bind is a property of the launcher, not a careless default. `start_servers.sh` now uses a `wait_healthy <url> <label> <max_attempts> <sleep_s>` helper with bounded retry (200×3s for clusters, 30×1s for tap/gateway) instead of `until curl … do sleep 3; done` — script exits non-zero on timeout, logs the cluster label, points at $LOG. `launch_vast.sh` `pip install` step now uses `python3 -m pip` with an `ensurepip` bootstrap fallback because the nix-based vast image may not have pip on PATH; on gateway-wait timeout, the launcher now `tail`s `/root/start.out` and the per-server logs over SSH instead of silently exiting. Dropped the dead-code conditional `if runner_path: env["RUNNER_MODEL_PATH"] = runner_path` in both cluster shims (`os.environ.copy()` already carries it). Skipped four lower-priority items by reviewer/tester consensus or explicit plan decision: 400-vs-422 distinction (client doesn't care, already 400), `/verify` 4xx-on-bad-sig (plan §3 explicitly chose to keep 200 with `is_verified=false`), ns-resolution alarm timestamps (tester said "fine for the demo"), `/tmp` EACCES on shared single-user box (cosmetic). Re-ran 14 envelope unit tests + CPU 4-server `--mock` smoke after every change: all pass; happy path returns `MOCK[hi|max=8]`, no alarm written. — Built all of plan §9 in `demos/tap-protocol/`: the c3-correct manifest `qwen3-1.7b-tap.manifest.json` (validates against `manifest.v1.schema.json` and Pydantic `Manifest`; `_build_vllm_cmd` yields `--enforce-eager --gpu-memory-utilization 0.4 --max-num-seqs 8 --max-model-len 2048 --attention-backend FLASH_ATTN`), `servers/envelope.py` (HMAC over `canonical_json_bytes(data.model_dump())` — not Pydantic's JSON — so dict insertion order can't drift the signature; 32-byte hardcoded `HMAC_KEY` with assert + docstring owning the non-secret threat model; `next_id()` is a module-level lock+counter, also `_reset_id_counter_for_tests` for unittest), `gateway.py` / `tap.py` / `host_cluster.py` / `recomp_cluster.py` (stdlib `http.server` + `ThreadingMixIn`, matching repo style). The two cluster shims share a `ClusterState` with `is_warm` gated on (a) child `/health=200` poll then (b) one canned `/v1/chat/completions` warm-up before flipping their own `/health` to 200 — so the start_servers loop's `until curl` is a single signal for ready-to-serve. Boot work runs in a daemon thread so the HTTP server binds immediately. Recomp's `alarm.jsonl` is opened with `"a"` per record (lock-guarded) and writes one `canonical_json_text(record)` line; the `[ALARM]` stderr line names id+reason. `client.py` is stdlib-only `urllib.request` and exits 1 on HTTPError. Added `tests/unit/test_tap_envelope.py` with 14 cases (round-trip, tamper of signature/payload/id/empty, dict-order stability incl. nested, JSON-wire re-serialization, monotonic id sequential + 8-thread × 200 reqs concurrency invariant, key length, manual HMAC regression) — all pass under `.venv/bin/python3 -m unittest tests.unit.test_tap_envelope`. CPU smoke (`--mock` on host + recomp, all 4 servers on ports 18000/18010/18020/18030): client gets `MOCK[Mismatch test|max=64]`, recomp `--mock-output-override "DIFFERENT"` triggers the alarm path — `alarm.jsonl` got exactly one line containing `id=1`, both sha256s, and the prefixes, and the tap's async thread logged `is_verified: False` from its `/verify` call. Surprises/deviations: (1) The plan said gateway/tap files would just `from servers.envelope import …`, but `demos/tap-protocol` has a hyphen so it isn't a Python package — I added `sys.path.insert(0, DEMO_DIR)` at module top of each server file so direct `python3 demos/tap-protocol/servers/X.py` works. The test file does the same. No deviation from the wire/protocol contract. (2) The `--proxy-port` flag name (rather than `--vllm-proxy-port`) matches start_servers.sh exactly. (3) For pydantic, used `dict[str, Any]` for `payload`; pydantic accepts dict-typed BaseModel fields without further nesting, and the wire round-trip test confirms `SignedEnvelope.model_validate(json.loads(model_dump_json))` re-verifies.
